Symulacja Rynku Giełdowego — diagramy projektu

Zestaw diagramów dokumentujących program. Schemat główny pokazuje pętlę symulacji z rozgałęzieniami decyzyjnymi, schemat silnika „otwiera czarną skrzynkę” dopasowania zleceń, schemat agentów opisuje logikę każdej strategii, a diagram klas (UML) odwzorowuje rzeczywistą strukturę kodu (klasy w market_simulation.html).

Przejdź do: 1. Schemat główny2. Silnik dopasowania 3. Logika agentów4. Diagram klas (UML)

Legenda symboli: owal ([ ]) = start / stop  •  prostokąt [ ] = proces (operacja)  •  romb { } = decyzja (rozgałęzienie tak/nie)  •  prostokąt z paskami [[ ]] = podprogram (osobny diagram)  •  sześciokąt {{ }} = początek pętli  •  równoległobok [/ /] = wejście / zdarzenie  •  krawędź przerywana -.-> = zdarzenie asynchroniczne (poza pętlą).

1. Schemat główny — pętla symulacji

Jedna „tura” (co 250 ms): animator rynku uzupełnia płynność, agenci i użytkownik składają zlecenia, silnik dopasowuje je, następuje aktualizacja ceny/wolumenu, BID/ASK/spreadu oraz odświeżenie interfejsu. Zlecenie użytkownika trafia do pętli przez asynchroniczne zdarzenie z formularza (krawędź przerywana).

    Start([Start]) --> I1["Inicjalizacja: ksiega zlecen, cena=100.00,
historia, tick=0, animator rynku + agenci"]
    I1 --> I2["Utworzenie wykresow i interfejsu"]
    I2 --> LOOP{{"Petla symulacji — co 250 ms"}}

    subgraph TURA["Pojedyncza tura symulacji"]
      direction TB

      INC --> MM["Animator rynku: uzupelnij plynnosc
na drabinie poziomow (sciany)
liczba/glebokosc/spread wg profilu plynnosci"]
      MM --> A1["Dla kazdego agenta: wywolaj decyzje act(ctx)"]
      A1 --> ADEC{"Agent zwrocil zlecenie?"}
      ADEC -- nie --> AMORE{"Czy sa kolejni agenci?"}
      ADEC -- tak --> ATYPE{"Typ zlecenia?"}
      ATYPE -- "z limitem" --> ADD["Dodaj limit do ksiegi

      ATYPE -- "rynkowe" --> ENG[["Realizacja zlecenia rynkowego

2. Silnik dopasowania — „chodzenie po księdze”

Wspólna procedura dla zleceń agentów, wieloryba i użytkownika. Zlecenie rynkowe konsumuje kolejne poziomy przeciwnej strony księgi (cena „wędruje” do każdego poziomu = cena maklera). Zlecenie z limitem realizuje się tylko dopóki „krzyżuje” swoją cenę, a niezrealizowana reszta zostaje w księdze jako nowa płynność. 

flowchart TD
    M0(["Wejscie: zlecenie (rynkowe lub z limitem)"]) --> M1{"Strona = kupno?"}
    M1 -- "kupno (taker)" --> MB["ksiega = ASKS (oferty sprzedazy)"]
    M1 -- "sprzedaz (taker)" --> MS["ksiega = BIDS (oferty kupna)"]
    MB --> ML{"ilosc > 0 ORAZ ksiega niepusta?"}
    MS --> ML
    ML -- nie --> MLIM{"Zlecenie z limitem i zostala ilosc?"}
    MLIM -- tak --> MREST["Dodaj reszte do ksiegi
(zlecenie spoczywajace = nowa plynnosc)"]
    MLIM -- nie --> MEND(["Koniec — cena przeszla przez poziomy"])
    MREST --> MEND
    ML -- tak --> MX{"Limit: czy poziom nadal
krzyzuje cene zlecenia?"}
    MX -- nie --> MLIM
    MX -- "tak / rynkowe" --> MT["poziom = ksiega[0]
tradeQty = min(ilosc, poziom.ilosc)"]
    MT --> MP["cena = poziom.cena (cena maklera)
wolumen += tradeQty"]
    MP --> MR["poziom.ilosc -= tradeQty
ilosc -= tradeQty"]
    MR --> MRM{"poziom.ilosc <= 0?"}
    MRM -- tak --> MSHIFT["Usun poziom z ksiegi
(plynnosc skonsumowana)"]
    MRM -- nie --> ML
    MSHIFT --> ML

3. Logika agentów — strategie uczestników rynku

Każdy typ agenta ma własną regułę decyzyjną. Kluczowa dla realizmu jest przewaga zleceń rynkowych (takerów), które konsumują płynność i poruszają cenę, podczas gdy animator rynku rozkłada zlecenia z limitem na wielu poziomach — tak powstają ściany płynności (pamięć rynku = wsparcie/opór). 

flowchart TD
    subgraph MMK["Animator rynku (Market Maker)"]
      direction TB
      MM0([Tura]) --> MM1["Dla k = 0..(mmLevels-1):
dist = (mmInnerTicks + k)*krok
bid = cena - dist, ask = cena + dist
(cena = ostatnia cena transakcyjna)"]
      MM1 --> MM2{"Plynnosc na poziomie < cel (mmTargetDepth)?"}
      MM2 -- tak --> MM3["Dodaj / uzupelnij limit"]
      MM2 -- nie --> MM4["Pomin poziom"]
    end
    subgraph NS["Trader szumowy"]
      direction TB
      N0([Tura]) --> N1{"random < 0.5? (aktywny)"}
      N1 -- nie --> NX([Brak zlecenia])
      N1 -- tak --> N2["strona = losowo kupno / sprzedaz"]
      N2 --> N3{"random < 0.80?"}
      N3 -- tak --> N4([Zlecenie RYNKOWE — konsumuje plynnosc])
      N3 -- nie --> N5([Limit 1-3 poziomy od ostatniej ceny — pasywnie])
    end
    subgraph TF["Podazajacy za trendem"]
      direction TB
      T0([Tura]) --> T1{"Historia >= 5 punktow?"}
      T1 -- nie --> TX([Brak])
      T1 -- tak --> T2["momentum = suma 5 ostatnich zmian ceny"]
      T2 --> T3{"p = |momentum| * skala; random < p?"}
      T3 -- nie --> TX
      T3 -- tak --> T4{"momentum > 0?"}
      T4 -- tak --> T5["strona = kupno"]
      T4 -- nie --> T6["strona = sprzedaz"]
      T5 --> T7([Zlecenie RYNKOWE 90% — wzmacnia trend])
      T6 --> T7
    end
    subgraph WH["Wieloryb"]
      direction TB
      W0([Tura]) --> W1{"random < 0.015? (rzadko)"}
      W1 -- nie --> WX([Brak])
      W1 -- tak --> W2["ilosc 40-100;
strona z 60% biasem na trend"]
      W2 --> W3([Zlecenie RYNKOWE — duzy wplyw, wiele poziomow])
    end

4. Diagram klas (UML) — struktura kodu

Diagram odwzorowuje klasy faktycznie zaimplementowane w programie (nie model „życzeniowy”). Nazwy metod odpowiadają funkcjom w kodzie (executeOrder, addOrder, cancelStale, step), dzięki czemu dokumentacja i kod pozostają spójne. 

classDiagram
    class Order {
      +int id
      +string side
      +string type
      +int priceCents
      +int qty
      +int bornTick
      +bool isMM
    }
    class OrderBook {
      +List~Order~ bids
      +List~Order~ asks
      +addOrder(o)
      +bestBid() int
      +bestAsk() int
      +spread() int
      +mid() number
      +cancelStale(tick)
      +depthProfile(bin)
    }
    class MatchingEngine {
      +executeOrder(o) Result
      +clearCrossedBook()
    }
    class Agent {
      <<abstract>>
      +int id
      +act(ctx) Order
    }
    class MarketMaker {
      +replenish(ctx) List~Order~
    }
    class NoiseTrader
    class TrendFollower
    class Whale
    class Simulation {
      +OrderBook book
      +MatchingEngine engine
      +MarketMaker marketMaker
      +List~NoiseTrader~ noiseTraders
      +List~TrendFollower~ trendFollowers
      +Whale whale
      +int lastPriceCents
      +List~int~ history
      +int tick
      +step()
      +reset()
      +routeOrder(o)
      +submitUserOrder(o)
      +fireWhale(side)
    }
    class Renderer {
      +render()
      +_renderKPI()
      +_renderPrice()
      +_renderDepth()
      +_renderHeatmap()
      +_renderBook()
      +_renderStatus()
    }
    Agent <|-- MarketMaker
    Agent <|-- NoiseTrader
    Agent <|-- TrendFollower
    Agent <|-- Whale
    Simulation "1" *-- "1" OrderBook : zawiera
    Simulation "1" *-- "1" MatchingEngine : zawiera
    Simulation "1" o-- "*" Agent : steruje
    OrderBook "1" o-- "*" Order : przechowuje
    MatchingEngine ..> OrderBook : modyfikuje
    Renderer ..> Simulation : odczytuje stan
